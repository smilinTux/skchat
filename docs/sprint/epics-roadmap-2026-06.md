# skchat Backend Unified Roadmap

## Epic summaries

**U14 — adapter-reply-routing.** Target: an `AdapterHub` in skchat that receives inbound `ChannelMessage` from already-shipped skcomms adapters (TG/Discord/Slack/Matrix), writes to ChatHistory, fires AdvocacyEngine, and routes the reply back via `adapter.send()` — plus a `/bind <fqid>` CapAuth flow. Headline value: closes the only-direction-left gap — agents can finally **converse on external platforms**. AdapterRegistry/adapters are 100% unit-tested and shipped, so the inbound→reply loop is buildable now with FakeAdapter.

**U8 — Federation Discovery Client.** Target: a `FederationDiscoveryClient` (`spaces/federation/discovery.py`) that builds signed assertions, queries Nostr relays for memberships, runs the (already-tested) deterministic focus election, and POSTs to the LIVE `/sfu/get` to mint a cross-host LiveKit token; browser then joins the remote focus instead of local. Headline value: closes **federated browser join** (jarvis@.41 → Space on .158) — server token-mint leg is already LIVE, only the client is missing.

**U9 — glossa-mesh-live.** Target: daemon spawns a `GlossaMeshSession` per live Space; listener-join enrolls into `LiveKitBus`+`GlossaMeshNode`, `on_message` routes to advocacy+memory, `on_leave` un-caps, with capauth frame-signing (`GlossaMeshGatekeeper`) and human-readable audit gloss. Headline value: closes **agents negotiating SKGlossa densely over a real Space** while humans read the gloss. All mesh primitives are CI-proven with zero production callers.

**U11 — Streaming voice / SKVoice GPU.** Target: a lightweight immutable `Conversation` VO threaded per-turn into `VoiceEngine.respond()` → `ctx["convo"]`, unblocking the Phase-3 worship tool stubs (`worship_session/list/replay`) and a `VoiceSession` per-connection state holder. Headline value: gives worship tools the live Conversation context they're stubbed-waiting on, and tightens the **speak→STT→respond→TTS** loop. CI-green; only true GPU inference is gated.

## Dependency & sequencing

- **U14 is unblocked NOW.** AdapterRegistry + ChannelAdapter ABC + FakeAdapter are shipped and fully unit-tested. U14 Phase 1 (`AdapterHub` + daemon wiring + unit tests) has zero external dependency. This is the highest-value, lowest-friction starting point. Within U14: Phase 1 → Phase 2 (reply routing needs the hub) → Phase 3 (/bind, side-branchable from Phase 1-end) → Phase 4 (REST/docs, fully parallel).
- **U8 client is unblocked NOW** because its three reuse seams (assertion.py, focus.py, nostr_io.py) and the `/sfu/get` authd route are all LIVE/CI. `discovery.py` (Tasks 1.1–1.3) + unit tests build against fakes immediately. Browser wiring (Phase 3) depends only on `discover_and_elect()` existing. Live cross-host (Phase 5) is the only gated leg.
- **U9 first wave is 4 fully-independent files** (livekit_bus on_leave, gatekeeper, daemon_service stub, advocacy_bridge) — all primitives CI-proven. But the full daemon impl + spaces route are sequential after the wave, and live SFU (Phase 6) is gated.
- **U11 first wave is unblocked NOW** (Conversation VO has zero deps). Phase 2 worship + Phase 3 VoiceSession both wait on Phase 1; Phase 3 then parallels Phase 2. GPU inference (Phase 4) gated.

**Value-maximizing order:** start U14-P1 + U8-P1 + U9-wave + U11-P1 simultaneously (all isolated, all CI-only). U14 reply-routing first delivers a visible product win (external-platform replies); U8 client delivers the federation payoff; U9/U11 firstwaves de-risk their larger sequential phases. Defer all live/GPU/two-host legs (see §4).
