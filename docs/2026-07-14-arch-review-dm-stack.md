# SKChat/SKComms DM Stack — Definitive Architecture Assessment

## EXECUTIVE SUMMARY
1. **Is this architectural?** Yes, and four newly-mapped subsystems (message stores, responders, delivery rails, identity/auth) all show the *same* structural shape as the crypto/control-plane findings: for each concern a clean, owned component exists (`ChatTransport`, `PersistentOutbox`, the CapAuth call plane, `skcomms.paths`, the message-id field) but the **live path is an improvised bypass**, and there is **no single authority** for the log, the responder, delivery-confirmation, or identity. The pain is not a cluster of bugs; it is the absence of ownership at four boundaries at once. The keystone remains the god-handler `daemon_proxy.api_send`, which fuses crypto, inference, control-demux, and persistence and bypasses every one of those clean components.
2. **Highest-leverage refactor (availability axis):** get LLM inference and reply delivery off the synchronous request path — async `202` + a **durable** reply-job queue + real `/ws/chat` delivery (SEAM 1). It is the keystone the control-plane, message-id, presence, rail, and responder work all hang off. **The deeper structural fix underneath it** is to establish four single authorities that today do not exist: **one authoritative message log** (SEAM 2), **one responder per (agent, channel)** (SEAM 3), **one enforced identity on the data plane** (SEAM 7), and **one delivery-confirmation state machine** (SEAM 10).
3. **Rough effort:** P0 stop-the-bleeding ~1-3 days (now includes disabling duplicate responders and adding data-plane auth, both near-zero-risk); P1 async/durable-delivery + message-id + single-log ~3-5 weeks (a durable queue + versioned dual-serve + a single-writer log, not a `to_thread` wrapper); P2 crypto-boundary + client ratchet-session lifecycle ~3-5 weeks and is a partial rebuild; P3 control-plane ~1-2 weeks. Groups + at-rest sealing + root-key protection + data-plane authN/authZ + prekey verification + rail-layer state machine are named workstreams on top.
4. **Trust-boundary decision must be made before any crypto work:** the draft's "one boundary at the server transport, read API returns plaintext" **downgrades E2E to hop-by-hop over a public funnel** and abandons a built at-rest sealer. Corrected target (unchanged, and reinforced by the identity map): the encrypt/decrypt boundary stays at the **client edge**; the daemon relays sealed frames and opens only in-memory to feed the brain; bodies are **sealed at rest**.
5. **Do first:** four tiny changes that stop active harm — (a) make a missing key **fail CLOSED** (refuse/queue, never emit cleartext); (b) **intercept `__REACT__/__EDIT__`** so control frames stop triggering brain replies; (c) **disable the duplicate DM responders** (`skchat-lumina-bridge.service`, scope `jarvis-heartbeat` to `jarvis` only) so a single message stops being answered up to 3×; (d) **add fail-closed CapAuth to `/api/send` and `/api/v1/prekey`** so the chat plane stops accepting unauthenticated writes and unsigned prekeys — then build the **versioned dual-serve** window before touching the send contract at all.

## 1. CURRENT ARCHITECTURE

SKChat has, in effect, two separate messaging stacks welded to a **fragmented set of stores**, answered by an **emergent set of responders**, over a **rail layer the live app never touches**, with **identity enforced on one plane and faked on the other**. The client uses the improvised half of every one of those splits.

### 1.1 Two stacks, and the app uses the wrong one

There is a **clean, layered transport** (`ChatTransport.send`/`receive`, transport.py) that seals once at egress, opens once at ingress, stores local plaintext, and routes control signals out-of-band as skcomms typed envelopes. **Two facts change everything about how "clean" this path is in practice:**
- It is **only exercised by the CLI/daemon-poll path**, never the app.
- Its DM ratchet is **OFF BY DEFAULT.** `transport.py:325` gates the ratchet on `os.getenv("SKCHAT_DM_RATCHET")`; unset (`""`) disables it. So "just make the app use `ChatTransport`" is not a matter of routing — the reference path itself is disabled by default and must be turned on and hardened first.

The **app** talks over plain HTTP to a second, improvised stack centered on `daemon_proxy.api_send`, where encryption framing, LLM inference, control-signal demux, and persistence are crammed into one synchronous HTTP handler. That handler:
- seals with a **different wire scheme** (`pqdm1:` one-shot) than the transport (`pqdr1:` ratchet),
- runs a **blocking 180s model call inline on the shared event loop** (confirmed: `api_send` is `async def` at daemon_proxy.py:1074, calls blocking `brain.reply` at 1262 with no `run_in_executor`/`to_thread` anywhere in the module),
- string-sniffs `__SENTINEL__` prefixes to tell control from chat,
- and persists a direction-dependent mix of plaintext (inbound) and ciphertext (outbound) into the same store.

The app then compensates client-side for every seam: 4s polling instead of the dead `/ws/chat`, a volatile `recallOutbound` cache to render its own outbound (because it literally cannot decrypt messages it sealed to the peer's key), fuzzy content/time dedup because there is no stable message id, and a false-offline debounce to hide the event-loop stall. It only appears to work because Chef-app and Lumina share one daemon and one history DB on .158.

### 1.2 The store reality: two pipelines, ten stores, no superset

A single logical DM can be physically written into as many as **ten stores**, and there is **no store that is a superset of the others**:

| # | Store | Written by |
|---|-------|------------|
| 1 | JSONL per-day log `~/.skchat/history/YYYY-MM-DD.jsonl` | `history.py:107-111` (`save`) |
| 2 | skmemory SQLite index `~/.skchat/memory/` (plaintext content) | `history.py:517` (`store_message`) |
| 3 | skmem-pg (pgvector) RAG corpus (Syncthing-replicated .158/.41) | `bridge_consciousness.py:113-124` (`LiveMemory.store`) |
| 4 | skcomms loopback outbox `~/.skcomms/outbox/*.skc.json` (unencrypted) | `transport.py:107-147` (`_write_local_loopback`) |
| 5 | skcomms per-fingerprint file inbox `~/.skcomms/transport/file/inbox/<fp>/` | skcomms FileTransport (`transport.py:33`) |
| 6 | skcomms shared inbox `~/.skcomms/inbox/` | `lumina-bridge.py:924-996` (`deliver_reply_to_inbox`) |
| 7 | skcomms per-agent comms dirs `~/.skcapstone/agents/<a>/comms/{inbox,outbox}` | `skcomms/paths.py:131-138` |
| 8 | Syncthing comms root `~/.skcapstone/sync/comms/outbox/<peer>/` | `skcomms/transports/syncthing.py:175-176` |
| 9 | skcomms retry-outbox + queue (3 fallbacks) | `paths.py:233,246,264`; `outbox.py`; `queue.py` |
| 10 | App Hive cache `messages_<peerId>` (client) | `message_repository.dart:31-41` |

These are written by **two independent write-and-reply pipelines that never converge:**

- **Pipeline A — App / webui path (`daemon_proxy.api_send`, the god-handler):** `POST /api/v1/send` (daemon_proxy.py:1073) → persist user turn to **JSONL only** (`_persist`→`hist.save`, daemon_proxy.py:1221,148) → build context by *re-reading* the same JSONL it is concurrently appending to (`_lumina_messages`, daemon_proxy.py:353-377) → call `LuminaBrain.reply` **inline, synchronously** (daemon_proxy.py:1262) → persist reply to JSONL (daemon_proxy.py:1278) → return it in the HTTP response. **This path never writes stores 4-9 and never enqueues anything.** It is a closed loop over store 1, with skmemory recall *disabled* (`SKCHAT_DM_MEMORY=0`, daemon_proxy.py:106-111), so app DMs also skip store 3.
- **Pipeline B — CLI / MCP / transport path:** `transport.send_message` (transport.py:404) writes store 2 (`store_message`, :490), store 4 (`_write_local_loopback`), and hands to `skcomms.send`/`send_federated` (:514,535) → the **receive daemon** (`daemon.py:703 poll_inbox`) writes store 2 + store 1 on receive (transport.py:733,742), then dispatches to a responder; **and separately** `lumina-bridge.py` polls stores 4/5/6 and replies via `deliver_reply_to_inbox` → writing store 1 + store 2 + store 6 (lumina-bridge.py:948-996). The Telegram bridge is a **third** writer, hitting store 3 via `LiveMemory` (telegram_bridge.py:557) with its own in-process history, not reading skchat's JSONL at all.

Whether a message enters the agent's RAG corpus (store 3), or the SQLite index (store 2), or only the JSONL (store 1) depends on **which pipeline delivered it**, not on any policy. The corpus is a non-deterministic subset of the conversation.

### 1.3 The responder reality: emergent ownership, verified on the live systemd plane

Reply generation is **not** owned. On .158 the following units are `active`, and several answer the *same* `(agent, channel)`:

| Live unit | Reply engine | Reads | Owns |
|---|---|---|---|
| `skchat-webui@lumina` → `daemon_proxy.api_send` | `LuminaBrain`, inline | HTTP `/api/v1/send` | (lumina, app-HTTP) |
| `skchat-daemon` → `GroupResponder.respond_direct` | skgateway `sk-default` | `transport.poll_inbox()` | (lumina, skcomms-DM) |
| `skchat-lumina-bridge` (`lumina-bridge.py`) | consciousness / skgateway | `~/.skcomms/inbox/` + `outbox/` + `transport/file/inbox/` | (lumina, skcomms-DM) — **duplicate of respond_direct** |
| `jarvis-heartbeat` (`jarvis-heartbeat.py`) | **spawns a `claude` Claude Code session** | `~/.skcomms/inbox/` + memory DB | (jarvis + opus + **lumina**) — **third answerer** |
| `skchat-telegram-lumina`/`-opus` | qwen3.6 + own tool loop | Telegram `getUpdates` | (lumina/opus, TG) — clean single-owner |

The live daemon drop-in sets `SKCHAT_ADVOCACY_DISABLED=1` and `SKCHAT_GROUPS=…`, which disables `AdvocacyEngine` but **leaves `respond_direct` on** (daemon.py:489-491,610-624). So the SK inbox has **three** live owners (`respond_direct`, `lumina-bridge`, `jarvis-heartbeat`), each with its **own** process-local dedup and its **own** hand-maintained agent loop-breaker list (`group_responder._KNOWN_AGENTS` = 11 names; `lumina-bridge.KNOWN_AGENTS` = 15; `jarvis-heartbeat` = 3; `AdvocacyEngine` = none). This is masked today **only because the app's primary path (`/api/v1/send`) bypasses the inbox entirely** — the moment a DM lands in `~/.skcomms/inbox/` (via `/api/send`, `skchat send`, or a real remote peer) it is answered **up to 3×** by three different engines (qwen via skgateway, the consciousness cascade, and a spawned Claude Code session), at three different latencies and persona fidelities. There is even a second `api_send` in the *same* webui process (`webui.py:1064` `/api/send`, transport-relay, no brain) whose semantics differ from `daemon_proxy.py:1073` `/api/v1/send` (inline brain): whether a DM is answered once inline or relayed into the triple-owner inbox depends purely on which URL the client picks.

### 1.4 The identity reality: two planes, opposite postures

There are two auth planes over the *same* daemon, *same* peers, *same* CapAuth identities:

- **Call / signaling plane — fail-CLOSED, cryptographically enforced.** `skcomms/capauth_validator.py` verifies a `<FINGERPRINT>.<TS>.<SIG>` PGP bearer (±300s replay window, every failure path returns deny, capauth_validator.py:258-319); `signaling.py` binds the authenticated fingerprint, never the client-claimed `from` (signaling.py:196-226,418-423); call routes derive self from the canonical resolver `resolve_agent_identity().fqid` (call_routes.py:47-50) and drop any invite where `_verify.valid` is false (call_routes.py:184-186). The broker is fail-closed by default (`SKCOMMS_BROKER_REQUIRE_AUTH=1`).
- **Chat data plane — NO auth, identity hardcoded.** The only webui middleware is a debug logger (webui.py:44-75); `/api/send`, `/api/v1/prekey`, `/api/v1/inbox` carry **no bearer/token check anywhere** in `daemon_proxy.py`. App-path identity is a set of compile-time constants (`LUMINA_ID`, `OPERATOR_ID="chef@skworld.io"`, hardcoded fingerprint, daemon_proxy.py:40-45). Every inbound operator message is persisted as `OPERATOR_ID`, never the authenticated identity (daemon_proxy.py:1145,1192,1201,1222). Hybrid open/seal hardcode the parties (`sender_short="chef"`, `recipient="lumina"`, daemon_proxy.py:462-506). `/api/v1/identity` returns a literal `fingerprint: "active"` (daemon_proxy.py:572-576). The Flutter client mirrors the constant (`pqLocalShortProvider = 'chef'`, pq_conversation_service.dart:297) and publishes its prekey with `signature: null` and no `owner` (pq_prekey_service.dart:158-176). The operator's *actually-authenticated* CapAuth fingerprint (proven in `capauth_service.dart:115-156`) is **never propagated** into the message sender, the AAD parties, or the prekey owner.

### 1.5 The rail reality: the app bypasses the entire reliability machine

skcomms carries a real delivery-reliability stack — `Router` rail selection + failover/broadcast + store-and-forward (`router.py`), a single durable `PersistentOutbox` with retry/backoff (`outbox.py:11-27`), and an application `AckTracker` (`ack.py`). `ChatTransport` (Pipeline B) flows through all of it via `send_federated`. **The app's `daemon_proxy.api_send` bypasses `ChatTransport` entirely, so it also bypasses rail selection, ACK, and the durable outbox.** Today's delivery guarantees exist **only** on the `ChatTransport → SKComms` path, not on the god-handler app path. Any rail/ACK refactor is moot for the app until that bypass is removed.

### 1.6 Corrected crypto/identity facts (the original draft was materially wrong here — unchanged, and reinforced by the new maps)

- **The live app-path seal is UNSIGNED.** `_seal_hybrid_outbound`/`_open_hybrid_inbound` (daemon_proxy.py:462/485) are **thin wrappers over one `skcomms.pqdm` implementation** — the Python seal is single-sourced, not "4-5 parallel crypto implementations." They call `pqdm.seal`/`open_sealed` and perform **no signing**. Only the unused `ChatCrypto` variant (crypto.py:447) signs and binds `(sender, recipient)` into AAD. So the live confidentiality surface has **no origin authentication at all.** What *is* duplicated is framing/AAD/prefix/base64 logic, and the AAD hardcodes `lumina`/`chef` (daemon_proxy.py:477/501, caller passes `sender_short="chef"` at 1113), so the app path only works for the single Chef↔Lumina pair. **The identity map deepens this:** the AAD hardcodes *both* endpoints, so the downgrade-lock and party-authentication are **decorative on the app path** — they authenticate a constant, not the actual conversants (pq_conversation_service.dart:297; daemon_proxy.py:462,485).
- **The "observed session failure" was misattributed.** The app path never calls `load_agent_crypto` or touches the PGP key — it depends only on the hybrid keypair (`pq_prekeys.lumina_private()`/`load_peer_bundle()`). The real app-path silent degradation is: `_seal_hybrid_outbound` returns `None` on any missing prekey/backend, the caller keeps **plaintext**, and logs only at `debug`. The `load_agent_crypto`→`None`→ratchet-off failure is real but lives on the **CLI/transport path**, doubly gated (also by off-by-default `SKCHAT_DM_RATCHET`). Both are fail-open; different code paths; both must be fixed.
- **Persistence is plaintext at rest, and that is a deliberate, orphaning decision.** `history.py:488` documents "plaintext content is stored." A full at-rest sealer exists — `encrypted_store.py` + `atrest_wrap.py`, DEK sealed to the recipient hybrid pubkey, suite-tagged/versioned — **built and never wired into the live path.**
- **The history store IS the agent's memory/RAG corpus.** `history.store_message` converts each message into an SKMemory snapshot → skmem-pg (pgvector, semantically searchable, **Syncthing-replicated across .158/.41**). DM plaintext fans into a replicated, vector-searchable long-term memory. "Clear history" must cover SQLite + JSONL **+ skmemory/RAG + every synced node.** (The identity map adds the authorization consequence: because unauthenticated writes reach this corpus, they become durable agent "memory.")
- **The root hybrid private key is plaintext on disk.** `~/.skchat/pqc/<agent>_hybrid.key` is plaintext hex, protected only by `0600` (pq_prekeys.py). At-rest epoch sealing under HKDF of a key that sits in cleartext next to it, on a synced filesystem, is defense-in-name.
- **Groups have the identical defect and are unfixed.** `daemon_proxy_groups.fan_out_send` stores `content=content` (plaintext) while `group_ratchet.py` exists unused. Group messages are plaintext on wire and at rest.

## 2. STRUCTURAL SEAMS (ranked by leverage)

> **Ranking axis, stated explicitly:** the primary ordering below is **operational-availability-weighted** — optimized to stop user-visible breakage fastest. On a **security-weighted** axis the order changes materially, and the four newly-mapped subsystems sharpen this. Security-weighted, the top of the list is **SEAM 7 (chat data plane has no authentication at all; unsigned prekeys enable handshake MITM)**, then **SEAM 6 (no origin auth / plaintext at rest / memory fan-in)** and **SEAM 9 (unsealed groups)**; these outrank the latency and duplication defects. On the availability axis, **SEAM 1 (god-handler)** is the keystone, and **SEAM 2 (no authoritative log)** + **SEAM 3 (no responder owner)** are the substrate under the duplication/replay/desync the user actually feels. Both axes are legitimate; the operator should pick knowing SEAM 1/2/3 fix felt pain while SEAM 6/7/9 fix actual confidentiality and authorization.

### SEAM 1 — The app-facing send handler is a god-function fusing four layers that must be separate: transport crypto, LLM inference, control demux, persistence.
**The seam:** `daemon_proxy.api_send` (daemon_proxy.py:1074-1310) opens inbound crypto (1113), string-sniffs control (1129/1137), runs blocking `brain.reply` inline on the shared loop (1262 → bridge_consciousness.py:616 `urlopen(180s)`), re-seals (1275), writes ciphertext to history (1274-1285). It also **bypasses the entire rail-selection/ACK/durable-outbox machinery** (§1.5) and the **entire identity-auth plane** (§1.4). The correct boundary (`ChatTransport.send`/`receive`) is bypassed.
**Why it breaks things:** every concern shares one request lifetime and one asyncio loop. Presentation latency is bound to LLM latency; one send freezes `/health`, `/ws/chat`, and WebRTC signaling. Because the handler bypasses `ChatTransport`, none of today's delivery guarantees (retry, ACK, outbox) apply to app traffic.
**Bugs it explains (confirmed):** send timeouts (10s Dio vs 15-180s gen), false-offline banner, ciphertext blobs in brain context and UI, "clearing history didn't clear context," no durable retry on app sends.
**Caveat (was overstated):** the "~1300 WebRTC reconnects/sec" is **correlated, mechanism unproven.** A frozen loop *stalls* signaling; it does not *accept* 1300 reconnects/sec. The rail map's independent finding is that WebRTC's ~1000/s / 255k-log-line spin was a **short-session reconnect storm fixed by `_SIGNALING_MIN_HEALTHY_UPTIME=3s`** (transports/webrtc.py:53-56,525-561) — i.e. an independent client retry/backoff pathology, at most amplified by the stall. **Do not credit the frozen loop as the storm's cause.**

### SEAM 2 — There is no single authoritative message log; "source of truth" is a race between processes across ~10 stores. *(NEW — from the stores map)*
**The seam:** JSONL (history.py:107) is *documented* as "primary storage," but stores 2, 3, and 10 are written independently, and the JSONL itself is appended by **at least four processes** (webui daemon_proxy.py:148, receive daemon transport.py:742, lumina-bridge.py:949, guest/group routes guest_group_routes.py:412) holding **independent SQLite handles** (each `ChatHistory.from_config()` opens its own — daemon.py:410, daemon_proxy.py:79, lumina-bridge.py:948). Append-only JSONL tolerates concurrent appends, but the **in-place rewriters** (`update_message`, `set_reaction`, `edit_message`, `prune` — history.py:304-341,218-270) do read-all → rewrite-file **with no lock**; a concurrent append or a second rewriter **silently loses lines.** Delivery is achieved by **re-ingesting one's own writes**: `deliver_reply_to_inbox` writes the reply to JSONL *and* drops an envelope in `~/.skcomms/inbox/` (lumina-bridge.py:949,957-996) so the daemon re-reads and re-stores it (transport.py:733,742) — the same message written to store 1 twice by two processes. The spool has **~9 candidate inbox/outbox roots** with overlapping semantics: skchat hardcodes `~/.skcomms/*` (transport.py:28,33,60) while `skcomms/paths.py` computes *different* per-agent roots (paths.py:131-138,146-153,194,264-268), and paths.py's own docstring (7-18) admits these historically disagreed. The `_write_local_loopback` copy (transport.py:91) exists **solely** to paper over this — a second, unencrypted store created to compensate for a path-ownership failure.
**Why it breaks things:** there is no owner of the log and no concurrency contract; correctness depends on processes happening not to write at the same instant. `store_message` (SQLite index) and `save` (JSONL) are **two half-writes callers must remember to pair** (transport.py:734-742 carries a copy-pasted comment enforcing the invariant; `agent_comm.py:188`/`cli.py:400` call only one, `attachments.py:164`/`webui.py:1056` call only the other), so *which messages are searchable vs. visible is caller-dependent.* Reads are unbounded full-scans of the JSONL tree (history.py:140-177), so the "index" is effectively write-only for the chat UI.
**Bugs it explains:** silent message loss under concurrent edit/prune, duplicate stored turns from re-ingest, RAG corpus being a non-deterministic subset, "clear history" leaving copies on synced nodes.

### SEAM 3 — No single responder owns any (agent, channel); ownership is emergent from which units are enabled. *(NEW — from the responders map)*
**The seam:** three live processes poll overlapping inbox directories for `(lumina, SK inbox)` and each answers with **independent** dedup state: `skchat-daemon respond_direct` (daemon.py:610, group_responder.py:263), `lumina-bridge.py` (lumina-bridge.py:1091-1117, reads the same `~/.skcomms/inbox/`), and `jarvis-heartbeat.py` (`_detect_agent` matches any lumina variant, jarvis-heartbeat.py:243-256). There is **no shared message-id idempotency store**, so cross-responder single-delivery is structurally impossible. `lumina-bridge.py` is a **live legacy duplicate of `respond_direct`** (`enabled`+`active`, different brain, different soul-prompt path, different loop-breaker) that will drift persona and answer twice. `jarvis-heartbeat` answers with a **different substance** (a spawned Claude Code session) at very different latency/fidelity. The one ownership knob, `SKCHAT_ADVOCACY_DISABLED` (daemon.py:467-482), disables only `AdvocacyEngine` — not `respond_direct`, `lumina-bridge`, or `jarvis-heartbeat` — so it does not actually transfer ownership of the DM channel. A legacy `skcomm-daemon.service` (skcomm, not skcomms) is also `active` as a fourth un-owned poller.
**Why it breaks things:** the effective owner of `(agent, channel)` is the set of enabled units × the directories they scan, and loop-prevention is re-implemented in four divergent agent lists (a name present in one but absent from another is a ping-pong hole).
**Bugs it explains:** the inbox-path half of "2-3 replies per DM" (up to 3 engines answer one message), persona drift between qwen/consciousness/Claude-Code replies, message ping-pong between agents whose names disagree across loop-breaker lists. Masked today only because the app's primary path bypasses the inbox.
**Clean counter-example (in-stack):** the Telegram path — each bot is a distinct ingress with offset-based dedup and exactly one process (telegram_bridge.py:1059-1074). Single ingress → single owner → no overlap. That is the shape the SK-inbox side lacks.

### SEAM 4 — No end-to-end message-id authority; writers reconciled by fuzzy content/time matching.
**The seam:** UI state has three writers (optimistic tempId `now_ms`, inline send-reply, 4s poll) plus a native CLI that fabricates ids as `sender_timestampMs` (daemon_service.dart:35) that never match server ids. The loopback/bridge path parses the inner `ChatMessage` id but **falls back to the envelope UUID** on parse failure (lumina-bridge.py:742,753-755), and `deliver_reply_to_inbox` mints a **fresh** id for the reply (lumina-bridge.py:936-944) rather than carrying a server-assigned id. Reconciliation collapses to "same content + direction within ±10 s" (conversation_provider.dart:73-86, message_dedup.dart:19) — a key the code itself admits is lossy ("two genuinely identical sends collapse to one," message_dedup.dart:12-14). `/api/v1/send` has no idempotency token; the only guard is a process-local 90s content-dedup band-aid (daemon_proxy.py:1039).
**Why it breaks things:** duplicate suppression is probabilistic; a real double-send of "ok" and a phantom duplicate are indistinguishable because there is no authoritative id. This is the identity failure that makes SEAM 2's re-ingest and SEAM 3's multi-responder problems *visible* as duplicates. "Clear history" clears only the Hive box while server SQLite/JSONL **and skmemory** retain turns.
**Bugs it explains:** duplicate replies on timeout+resend, replays after "clearing history," own-outbound vanishing on restart.

### SEAM 5 — Control-plane rides the data-plane as `__SENTINEL__` content, demuxed by string prefix in the hot send path. *(was SEAM 3)*
**The seam:** `__REACT__/__TYPING__/__EDIT__/__RECEIPT__/__CALL_REQUEST__` and sealed `pqdm1:` ciphertext are sent as ordinary bodies through `/api/v1/send` (skcomms_sync.dart:228-325); the backend classifies by `startswith` (daemon_proxy.py:1068/1129/1137). The store has **no notion of message class**, so every reader re-implements a blocklist: `_is_context_noise` (daemon_proxy.py:1056-1070), the bridge's content-type skip (lumina-bridge.py:737), four client-side sentinel parsers (conversation_provider.dart:179-211), and a defense against `<event …>` CoT beacons (transport.py:740). skcomms already ships the right thing — a typed `MessageType` envelope with `make_read_receipt`/`make_ack` (models.py:36-42/194-227) — and the native transport uses it correctly; the app path ignores it.
**Why it breaks things:** the demux is duplicated and internally inconsistent — `__TYPING__/__RECEIPT__` are intercepted but `__REACT__/__EDIT__` fall through to `brain.reply` (daemon_proxy.py:1121-1128 names this as the root cause of the historical "2-3 unrelated replies"). A missed prefix (each control ping has unique content) becomes a spurious stored "turn," a spurious reply, **and a spurious RAG row.** Control traffic thus leaks into the store and the agent's memory.
**Bugs it explains:** the app-path half of "2-3 unrelated agent replies per DM" (Lumina brain-replies to a `__REACT__`/`__EDIT__` ping — **fix is interception, not context-noise filtering; see §4 P0**), typing broken every direction, receipts invisible across daemons, and app-vs-backend call paths that cannot ring each other (unsigned `__CALL_REQUEST__` body vs signed `CALL_INVITE`).

### SEAM 6 — No single crypto boundary; two incompatible DM wire schemes; and no origin authentication on the live path. *(was SEAM 2)*
**The seam:** `pqdm1` (one-shot) lives in `skcomms.pqdm`, wrapped thinly in `ChatCrypto` (crypto.py), in the web handler (daemon_proxy.py:462/485, hardcoding `lumina`/`chef` into AAD), and again in Dart. The transport seals with `pqdr1` (ratchet, dm_session.py), which the app codec does not recognize (`isHybridToken` matches only `pqdm1`).
**Corrections vs draft (unchanged):** the **Python seal is single-sourced** in `skcomms.pqdm` — the duplication is **framing/AAD wrappers**, not five crypto codepaths. The real crypto risk is **divergent AAD binding** (the hardcoded pair) plus **no signature on the live path.** Do **not** claim "pqdm1 signs" — the app's `_seal_hybrid_outbound` does not sign.
**Why it breaks things:** which scheme a message gets is decided by *which code path sent it*, not by the conversation; the app keeps a shadow `recallOutbound` map because it cannot decrypt its own outbound.
**Downstream confidentiality findings:** DM bodies are **plaintext at rest** (history.py:488) despite an **orphaned at-rest sealer** (`encrypted_store.py`/`atrest_wrap.py`); plaintext **fans into replicated pgvector memory** (`store_message`→SKMemory); and the **root hybrid key is plaintext on disk.**
**Bugs it explains:** undecryptable blobs when a `pqdr1` frame reaches the app, raw `pqdm1:` base64 in UI (read API mislabels `is_encrypted=False`, daemon_proxy.py:208/249), own-outbound vanishing on restart, the pair-hardcoded copy only working Chef↔Lumina.

### SEAM 7 — The chat data plane has NO identity authentication or authorization; app-path prekeys are unsigned and unverified, enabling handshake MITM. *(NEW — from the identity map; security-weighted #1)*
**The seam (two joined findings):**
- **No authN/authZ on the data plane.** `webui.py:44-75` is the only middleware and authenticates nothing. `/api/send` attributes every message to `OPERATOR_ID` (daemon_proxy.py:1192,1222); `/api/v1/prekey` accepts any bundle for any `owner` (daemon_proxy.py:509-529). **Anyone who can reach the webui port can post as chef, publish chef's prekey, and read chef's inbox** — and because history fans into skmem-pg, those unauthenticated writes become durable agent memory. The call plane proves the team can do fail-closed CapAuth (`capauth_validator.py`); the messaging plane simply never adopted it.
- **Unsigned + unverified prekeys → key-substitution / MITM on the PQ handshake.** The app sets `signature: null` (pq_prekey_service.dart:158); the daemon stores whatever is POSTed **without verification** (daemon_proxy.py:526 → pq_prekeys.store_peer_bundle, pq_prekeys.py:72-87); `_seal_hybrid_outbound` seals Lumina's replies to the *stored* bundle with **no signature check** (daemon_proxy.py:493-503). The ratchet path already has the correct control — `prekey_sig.verify_prekey_bundle(bundle, identity)` is enforced in dm_manager.py:141-143 and prekey_exchange.py:168-170 — but the app path is a **second, divergent implementation that dropped the authenticity check.** An attacker who POSTs a prekey under `owner=chef` (no auth, per above) gets **every subsequent Lumina reply sealed to their key.** The PQ layer's confidentiality is only as strong as the (here, absent) prekey authentication.
**Why it's structural, not a bug:** the entire chat plane was built with identity as a *presentation constant* (daemon_proxy.py:40-45; Flutter `'chef'` at pq_conversation_service.dart:297) rather than an *enforced dependency*. Three uncoordinated sources of truth coexist — the canonical capauth resolver (used by agents/CLI/calls), the daemon_proxy constants, and the Flutter constant — and the operator's actually-authenticated fingerprint is never propagated into any of them. `/api/v1/identity` even returns a fake `fingerprint: "active"` (daemon_proxy.py:572-576) that the Flutter `CapAuthService.login()` reads as its identity source (capauth_service.dart:115-156) — a route/contract collision (real fingerprint vs literal `"active"`) on the identity source itself.
**Bugs/exposures it explains:** impersonation of chef over the funnel, MITM decryption of agent replies via substituted prekeys, poisoned RAG memory from unauthenticated writes, the app path being structurally incapable of representing Opus↔chef or lumina↔jarvis without editing constants in two repos.

### SEAM 8 — Identity/crypto fails OPEN, silently, on multiple paths. *(was SEAM 5)*
**The seam:** the fail-closed crypto *core* (`pqdm.py` raises on missing backend; `DowngradeDetected` on tamper, pqdm.py:305-309,366-372) is wrapped by integration code that fails **open**. (a) CLI/transport: `load_agent_crypto` returns `None` on any failure (crypto.py:814-825), `transport.py:325` gates the ratchet on truthy crypto **and** the off-by-default `SKCHAT_DM_RATCHET`, so a missing PGP file (or unset flag) downgrades to plaintext with a debug log. (b) App path: `_open_hybrid_inbound` returns `None` when `PQ.lumina_private()` is missing and the caller keeps the ciphertext-as-content going plaintext to brain + history (daemon_proxy.py:469-471,1112-1116); `_seal_hybrid_outbound` returns `None` on any error and the caller ships plaintext (daemon_proxy.py:485-506). A missing/rotated `lumina_hybrid.key` degrades an advertised-PQ conversation to cleartext **with no operator-visible signal.**
**Why it breaks things:** the security posture flips at a layer boundary — the primitive says "never silently downgrade," the integration silently downgrades. The confidentiality surface has multiple single points of failure with **no health signal and no refusal.** A banner is not a fix; the correct posture is **fail-closed** (see §4).

### SEAM 9 — Group messaging carries every DM defect, unsealed. *(was SEAM 6)*
**The seam:** `daemon_proxy_groups.fan_out_send` stores plaintext (`content=content`) while `group_ratchet.py` sits unwired. Group messages are plaintext on wire and at rest, unauthenticated, and fan into the same replicated memory. This is a larger live exposure than any single DM defect and must be an explicit workstream, not silently scoped out.

### SEAM 10 — Delivery reliability lives in a rail layer the app bypasses, and even there confirmation truth has no single owner. *(NEW — from the rails map)*
**The seam (two joined findings):**
- **The app bypasses the machinery (see §1.5).** All retry/ACK/outbox guarantees exist only on the `ChatTransport → SKComms` path; the god-handler has none.
- **Even on the transport path, "did the peer receive it?" has no single owner.** There are **two competing confirmation authorities**: rail-`confirmed`/`queued_only` at the transport layer (transport.py:87-106) and peer-`ACK` at the application layer (ack.py). They are joined only for the narrow `queued_only ∧ ack_requested` case (core.py:985-1039), and **rail-`confirmed` wins by omission** — a `https-s2s` `{"ok":true}` send is not held in the outbox, so if its ACK never returns it **times out silently with no retry and no dead-letter** (core.py:916-917,1082-1088). The entire ACK lifecycle (`sweep_ack_timeouts`, `process_ack`, `_send_auto_ack`) runs **only inside `receive()`** (core.py:1209-1235), so the 300s wall-clock ACK guarantee is parasitic on inbound polling cadence. ACKs are also routed on a **different rail class** than the message they confirm — `_send_auto_ack` uses the legacy unsigned `route()` (core.py:1311), which *excludes* `https-s2s` (router.py:87,652), so a DM delivered over `https-s2s` cannot be ACKed over it. Durable retry is the **caller's** job, unenforced at the router boundary (router.py:298-302; only `core.SKComms` enqueues, core.py:882,947).
**Why it's structural:** the router hardcodes transport-specific policy by literal rail name (`POINT_TO_POINT_TRANSPORTS`, `FEDERATION_DEFAULT_CHAIN`, `SIGNED_ENVELOPE_ONLY_TRANSPORTS`, `DEFAULT_STORE_FORWARD_TRANSPORT` — router.py:42,66-74,87,97), two ordering authorities contradict (numeric `priority` vs the named chain; `webrtc` is `priority=1` but absent from the chain so it is tried **last**, router.py:694-712), and WebRTC's async warm-up is modeled as a synchronous pass/fail rail, so three warm-up misses push a rail that is *actively becoming usable* into a 60s cooldown (transports/webrtc.py:352-368; router.py:33-34,1168-1169). Most of the router's mass (cooldown, perm-backoff, quarantine, log-dedup) is **failure-accounting compensating for the absence of a live rail-readiness signal** (`is_available()` = "signaling up"/"dir writable", not real reachability).
**Why it's lower on the availability axis for *this* stack:** the app doesn't touch any of it today, so it is moot for the app until SEAM 1's bypass is removed — but it is the **reliability substrate for federation** (the stated justification for the whole effort), so it must be fixed before human↔human traffic depends on it.
**Clean, enforced parts (do not touch — see §5):** `PersistentOutbox` as the single durable queue of record (outbox.py:11-27; the two historical overlapping stores were deleted), and the signaling fingerprint binding (signaling.py:418-479, fail-closed broker default).

## 3. TARGET ARCHITECTURE

**The one decision that gates everything: where is the crypto boundary?** The draft put it at the **server transport** and had the read API return plaintext. That **downgrades end-to-end to hop-by-hop**: `ChatTransport` seals daemon↔peer, but the Flutter app is a *client of the daemon over a public Tailscale Funnel*, not a transport peer. Returning plaintext on read ships cleartext across the public edge and re-bakes the single-trust-domain assumption the whole refactor is meant to kill for federation. **Corrected target: the encrypt/decrypt boundary stays at the client edge.** The identity map reinforces this: the same daemon already terminates a fail-closed CapAuth boundary for calls, so the messaging plane should adopt the identical boundary rather than invent a plaintext-read shortcut.

- **Chef↔Lumina (human↔agent):** the agent must read plaintext in-memory to run the brain, so there is no client↔client E2E to preserve *on the brain leg*. But app↔daemon still crosses the public funnel, so the app **seals to the daemon** and the daemon opens **only in-memory** to feed the brain; the reply is **re-sealed to the app** before it leaves the daemon. The store holds **at-rest-sealed** bodies (via the existing `atrest_wrap`), not disk plaintext.
- **Human↔human (federation, the stated justification):** the daemon stays a **relay** and never becomes a plaintext waypoint. True client↔client E2E.

**One authoritative message log, single-writer, sequence-numbered.** Collapse SEAM 2's ten stores to a spine: **one append-only, sequence-numbered message log per conversation, owned by a single writer** (a daemon-owned SQLite/WAL table or durable queue), with a **server-assigned immutable `message_id` + monotonic `seq`.** JSONL becomes an *export*, not the store. **Persist-then-publish, one owner:** the daemon writes the log, then emits a delivery event; nothing else writes the log. Delivery is a **projection over the log** — the spool carries `(conversation_id, seq, id)` references and receivers upsert by id; **no loopback-copy store, no re-ingest-your-own-write.** Collapse the ~9 spool roots to the single `skcomms.paths`-resolved pair and make skchat use it (retiring the `_write_local_loopback` compensator). The index (SQLite/pg) becomes a **derived projection keyed by seq**, so search and RAG are reproducible and recoverable by replaying the log, never a divergent parallel write. `store_message`+`save` collapse into one write behind the log owner (no more paired half-writes).

**One responder per (agent, channel) via an owner table + a durable queue.** Declare ownership **centrally** in an owner table keyed `(agent, channel) → exactly one responder`, and **enforce it at startup** (refuse to run two responders that claim the same key) — ownership must not be emergent from unit enablement. Give each ingress a **single dispatcher** that unseals, demuxes control-plane, applies one canonical loop-breaker (from the identity resolver, not four hand-maintained lists), dedups on a **shared `message_id` store**, then hands a clean `(agent, context, text)` to the owning responder; responders become pure `generate(context) -> text`. `brain.reply` moves out of the request handler into a **durable, crash-recoverable reply-job store** (not `asyncio.to_thread`, which dies with the process — on a stack that restarts often from OOM caps/freezes, a 202 followed by a restart = **silent reply loss with a success already returned**). **Retire the duplicates now:** stop/disable `skchat-lumina-bridge.service` (its function *is* `respond_direct`), leave `opus-bridge.py` undeployed, and scope `jarvis-heartbeat` to `jarvis` only (or make it the *declared* owner for the agents it handles). Collapse the dual `webui.api_send`/`daemon_proxy.api_send` into one path. The Telegram single-owner pattern is the target shape.

**Identity as an enforced, propagated, fail-CLOSED dependency on the data plane.** Adopt the CapAuth boundary the call plane already proves: **add fail-closed authentication to `/api/send` and `/api/v1/prekey`** (reuse `CapAuthValidator`), **verify prekey signatures on the app path** (reuse `verify_prekey_bundle`, and require the app to *sign* its bundle — no more `signature: null`), and **propagate the authenticated CapAuth fingerprint** into the message sender, the AAD parties, and the prekey owner — deleting the `daemon_proxy`/Flutter identity constants and the fake `/api/v1/identity` fingerprint. A missing key does not silently downgrade: on send into an encrypted conversation with no key, the daemon/client **refuses or queues** and never emits cleartext; the `/health` flag is **secondary telemetry, not the mitigation.** Decouple confidentiality from signing (a missing *classical* PGP key may disable *signing* but must not disable *confidentiality*). Root key handling is upgraded from plaintext-`0600` to OS keyring / passphrase / capauth-sealed storage. This collapses SEAM 7's three identity authorities back to the one already declared canonical (`capauth.resolve_agent_identity`).

**Envelope-typed control plane, signed, and where linkable, sealed.** Typing/receipts/reactions/edits/call-signaling become **typed skcomms envelopes** delivered over the push channel, never message content — the store filters by **type**, not by string prefix, and control traffic is never a candidate "turn" (killing SEAM 5's leak into store/RAG). Control envelopes must be **signed** (reuse the capauth `CALL_INVITE` signing path) and linkable metadata **sealed.** Presence has one owner (`PresenceCache` fed by HEARTBEAT) behind a real endpoint replacing the `/api/v1/presence` no-op (daemon_proxy.py:1418). Prefix-sniffing is then **deleted**, not extended.

**Message-id and idempotency, authenticated.** Every log row carries the server-assigned `message_id` + `seq` and echoes a **client-supplied `client_dedup_key`** bound into the sealed frame's AAD (so it cannot be forged to suppress/censor a genuine message via key collision). This is a **precondition** of the async cutover and the shared cross-responder idempotency store.

**One delivery state machine, tiered confirmation, capability-driven selection (SEAM 10).** Model delivery as `queued → sent → rail-acked → peer-acked → dead`, owned by a single `DeliveryService` wrapping router+outbox+ack; rail-`confirmed` and `PendingAck` become **tiers of one state**, not two parallel truths (closing C1/H4). Replace the router's name-frozensets with `Transport` capability flags (`requires_signed`, `is_point_to_point`, `can_fan_out`, `is_store_forward`, `warmup=async|sync`, `confirms_receipt`) selected by capability + a live `readiness()` score (fixing the WebRTC warm-up mis-scoring and the vestigial `priority`). Move `sweep_ack_timeouts`/auto-ACK onto their **own scheduler** decoupled from `receive()`, route ACKs over the **same rail class** as the message, and express store-and-forward as the final **tier** of an ordered `direct → relay → deferred` policy that can hold ≥2 rails (honoring "get two"). This machinery is a prerequisite for federation, not for the Chef↔Lumina app leg.

**Async delivery, dual-served.** `POST /api/v1/send` returns **202** with `{message_id, client_dedup_key}`, idempotent, **only behind a versioned/capability handshake** so old clients keep the synchronous `{reply:{…}}` contract until telemetry shows they are gone. Reply/typing/receipts arrive over `/ws/chat` (or SSE). **Polling stays primary until WS delivery is measured reliable** over the funnel + Tailscale reconnect behavior, then demotes to a cold-start/reconnect backstop.

**At-rest and memory-fan-in are in scope.** Wire the orphaned `atrest_wrap` sealer into the (now single) log write path so bodies are sealed on disk/backups/synced nodes. Treat "clear history" as covering the log + JSONL export + **skmemory/pgvector + replicas.** Decide explicitly whether DM content should enter long-term agent memory at all, and gate it — the corpus should be a *policy-defined* projection of the log, not a pipeline-dependent accident.

### P2 — Confine crypto to one client-edge boundary and one scheme (SEAM 6) + propagate real identity (SEAM 7) — partial REBUILD, not a codec patch
1. **Decide the trust boundary first** (server-terminated vs client E2E). The target chooses **client-edge**; commit to it in writing.
2. **Build the Dart ratchet-session lifecycle.** This is **not** "teach the codec to recognize `pqdr1`." The app has **no ratchet session** — only stateless `pqdm1` one-shot. `pqdr1` needs prekey exchange (now **signature-verified**, per P0.5), an epoch-0 hybrid-KEM handshake, at-rest-sealed epoch state on the client, and **per-conversation negotiated cutover.** Keep the Dart **open** capability (do not "delete the Dart crypto" — that contradicts E2E delivery). **Propagate the authenticated CapAuth fingerprint into the AAD parties** (retiring the hardcoded `lumina`/`chef`), so the party-binding stops authenticating a constant.
3. **Migration is partly impossible — say so and pick a policy.** Existing **outbound rows are `pqdm1` sealed to the peer's key; the daemon cannot decrypt its own historical outbound.** Choose **quarantine behind a legacy-decode shim** (preferred) or accept data loss. Sequence: **decrypt-all → re-key/re-store → verify counts → shadow-read window → only then delete openers.** Keep **dual-open indefinitely** for archived history.
4. **Deletes are release-gated on client rollout, not server-local.** `_open_hybrid_inbound` cannot be removed while any deployed app still POSTs `pqdm1:` tokens.
5. **Honest read metadata, sealed frame.** Fix the `is_encrypted=False` lie (daemon_proxy.py:208/249) by **labeling truthfully and returning the sealed frame**, not by stripping ciphertext to plaintext.
6. **Wire the standardization to SEAM 6's convergence decision** below (trust-model tradeoff). *Risk:* high (crypto correctness + live-data migration + client rewrite). **Rebuild (Dart session) + confine-and-delete (server framing) + migration project.**

**Wire-scheme convergence is a trust-model decision, not a cleanup.** Standardizing on `pqdr1` **drops sender authentication and identity AAD binding** (`dm_manager` defaults `AuthMode.ANONYMOUS`; the open path does no signature verification; frame AAD is only `(epoch,index)`). `pqdm1` (the signed `ChatCrypto` variant) binds `(sender,recipient)` and signs the plaintext. Convergence trades **authenticated** for **deniable.** That may be defensible, but it must be an **explicit, argued decision** with the deniability property proven, and a **SOVEREIGN/signed mode** available for peers who need attributable messages. (This is now doubly load-bearing: with real identity propagated per P0.5/P2.2, a SOVEREIGN mode is the only way attribution survives.)

### P3 — Real, signed control-plane and presence authority (SEAM 5), additive-first
1. **Add typed skcomms envelopes end-to-end** over `/ws/chat`, **before** deprecating any sentinel. **Sign** control envelopes (reuse capauth `CALL_INVITE` signing) and **seal** linkable metadata. Then delete prefix-sniffing entirely (not grow the branch list) — the single-log store now filters by **type**, so control frames can never become a stored turn or a RAG row.
2. **Call signaling: additive-then-deprecate.** Ship signed `CALL_INVITE` emit/verify in the app **first**; only then retire `__CALL_REQUEST__`. **Prerequisite:** the app must obtain and **pin peer capauth public keys** to verify signatures (the same key-distribution work P0.5's prekey verification needs — do them together).
3. **Cross-daemon receipts are a FEDERATION project, not wiring.** `make_read_receipt` across daemons requires retry, ordering, and offline-peer delivery — a distributed-systems workstream that exists only because the target deliberately breaks the "one daemon on .158" assumption. **This is the natural home for the SEAM 10 delivery-state-machine work** (tiered confirmation, capability-driven selection, independent ACK clock), since federated receipts are exactly the rail/ACK path the app has so far bypassed.
4. Presence: give `PresenceCache` a real endpoint replacing the `/api/v1/presence` no-op (daemon_proxy.py:1418). *Risk:* medium-high. **Refactor-in-place + one federation build.**

### P-Sec (parallel, security-weighted — do not let the availability track bury these)
- **Data-plane authN/authZ + prekey verification (SEAM 7):** enforced in P0.5/P0.6; complete the fingerprint propagation in P2.2. This is the single highest-value security workstream.
- **Groups (SEAM 9):** wire `group_ratchet` into `fan_out_send`, seal per-member or via sender-key. Named workstream.
- **At-rest sealing:** wire `atrest_wrap`/`encrypted_store` into the single-log write path so bodies are not disk plaintext on synced nodes/backups.
- **Root-key protection:** move `<agent>_hybrid.key` off plaintext-`0600` to keyring/passphrase/capauth-sealed.
- **Memory fan-in governance:** make the RAG corpus a **policy-defined projection** of the single log, not a pipeline-dependent write; gate/scrub `store_message → SKMemory`; make "clear history" cover skmemory + replicas.
- **Rail-layer state machine (SEAM 10):** land the `DeliveryService` + capability-driven selection alongside the P3.3 federation work; moot for the app until SEAM 1's bypass is removed.

### Rollback (required on every high-risk tier)
Every P1/P2 tier ships behind a **feature flag with dual-write and a documented revert.** You cannot fall back to a blob-serving read API once you've written at-rest-sealed or plaintext-uniform rows and removed the old path, and you cannot un-collapse the store once the single-writer log is authoritative — so the revert must be designed *before* the cutover.

## 5. WHAT NOT TO TOUCH (these are correct)

- **`ChatTransport.send`/`receive` internals (transport.py:430-442/707-717)** — the reference layering (seal once, open once). **But it is off by default (`SKCHAT_DM_RATCHET`) and seals daemon↔peer, not client↔peer** — so the plan is "adopt this layering at the client edge and turn the ratchet on," not "rewrite the internals."
- **The `pqdr1` ratchet cryptographic internals** — `DmSession.seal`/`open`, `dm_ratchet` hybrid x25519-mlkem768 encap/decap, at-rest epoch sealing (dm_store.py). Cryptographically sound; the problems are **reach, default-off, missing sender auth, and a plaintext root key**, not the primitive.
- **The prekey-authenticity control that already exists on the ratchet path** — `prekey_sig.verify_prekey_bundle` enforced in dm_manager.py:141-143 / prekey_exchange.py:168-170. Do not rebuild it; **make the app path call it** (SEAM 7 / P0.5). This is the correct control the divergent app path dropped.
- **skcomms typed control-plane** — `MessageType` enum + `make_read_receipt`/`make_ack` (models.py:36-42/194-227) and the native HEARTBEAT→PresenceCache path. Adopt it; add **signing/sealing** on top; don't rebuild it.
- **The Telegram single-owner responder pattern** — one ingress (`getUpdates` offset), one process, one dedup (telegram_bridge.py:1059-1074). This is the ownership shape SEAM 3's fix should copy for the SK inbox. Don't touch it; **replicate it.**
- **`PersistentOutbox` as the single durable queue of record** — the two historical overlapping retry stores were deleted and the router genuinely persists nothing (outbox.py:11-27, router.py:273-275). Correct and enforced; the only gap is that *enqueue* still depends on the caller (SEAM 10/H4). Keep the store; move enqueue behind the delivery boundary.
- **Signaling identity binding** — the token-verified fingerprint is authoritative; the relay ignores the client-supplied `?peer=`/`from` except under an explicit dev gate; fail-closed broker default (signaling.py:418-479, broker_server.py:30-33). This seam is correct — it is the model the chat data plane must adopt (SEAM 7).
- **Backend call signaling** — capauth-signed `CALL_INVITE` with signer/from_fqid spoof-check (call_routes.py:184-209, call_session.py:37-62). The good identity path; converge the app onto it (with peer-key pinning).
- **The canonical identity resolver** — `identity_bridge.py` as a thin delegate to `capauth.resolve_agent_identity` (identity_bridge.py:46-88). Well-factored and honest about being a consumer; the fix is to **route the messaging plane through it** and delete the constants (SEAM 7), not to change the resolver. (One caveat: tighten `resolve_peer_name`'s URI-synthesis fallback ladder, identity_bridge.py:184-191, which can mint unauthenticated identities — belongs in the resolver, not a string-fallback.)
- **Daemon-held signing** — private key never leaves the daemon, `/api/v1/sign` returns armored sig. Correct trust boundary; add the missing degradation signal and fail-closed behavior.
- **`ChatCrypto`'s idempotency guards and ratchet wrappers (crypto.py:369/503/528/547)** — fine as internal implementation behind one boundary. This is the **signed** variant the app does *not* use; if signed/attributable DMs are wanted, route through it, don't delete it.

**Rebuild is warranted in exactly one place, and it is bigger than the draft claimed:** the Dart client's DM crypto is not a `pqdm1`→`pqdr1` codec swap — it is a **full ratchet-session lifecycle** (signature-verified prekey exchange, epoch-0 handshake, client-side sealed epoch state, per-conversation negotiated cutover). Everything else is **confine, connect, consolidate, delete-duplicates** — the correct components exist (`ChatTransport`, the single `PersistentOutbox`, the CapAuth call plane, `verify_prekey_bundle`, `skcomms.paths`, the Telegram single-owner pattern, the `message_id` field). The failures are that the good paths are disabled by default, reached by the wrong layer, un-owned across processes, missing origin auth *and* endpoint auth, storing plaintext at rest, replicating that plaintext into memory, and leaving groups, the root key, and the entire data-plane authorization boundary unsealed.

**Do NOT treat these claims as settled:** the "~1300 WebRTC reconnects/sec caused by the frozen loop" (the rail map attributes the spin to a short-session reconnect storm fixed by `_SIGNALING_MIN_HEALTHY_UPTIME`, not the loop stall — verify before betting P1 on it) and "the app has zero `/ws/chat` consumers" (map-sourced; app repo absent from this checkout — confirm before scoping P1c).
