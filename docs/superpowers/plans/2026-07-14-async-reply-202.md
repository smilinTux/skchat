# Flag-Gated Async Reply (202) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use `- [ ]` checkboxes.

**Goal:** Behind `SKCHAT_ASYNC_REPLY` (default OFF), `/api/v1/send` persists the user message, returns HTTP 202 immediately, and generates Lumina's reply in a background task that seals + persists it and pushes it via the existing `/ws/chat` "new" channel the app already handles. Default OFF = today's synchronous behavior, byte-for-byte. Fixes the timeout→retry→duplicate root cause (the app no longer waits ~180s on the request).

**Architecture:** Extract the existing reply-generation block from `api_send` into one coroutine used by BOTH paths. Sync path awaits it and returns the reply (unchanged). Async path schedules it via `asyncio.create_task` and returns 202. The reply already reaches the app via `_ws_broadcast({"type":"new"})` (webui.py) → the app refetches, so no app change is needed to RECEIVE the async reply.

**Tech Stack:** Python 3.10+, FastAPI, `daemon_proxy.py` (`api_send`), `asyncio`, existing `_persist`/`_seal_hybrid_outbound`/`_get_brain`/`_ws_broadcast`.

## Global Constraints

- PYTHON ONLY. `SKCHAT_ASYNC_REPLY` default OFF (truthiness convention matches `_seal_groups_enabled`). Flag OFF must be behavior-identical to today.
- Do NOT change the control-frame short-circuits (`__TYPING__`/`__RECEIPT__`/`__REACT__`/`__EDIT__`), the dedup band-aid (`_SEND_LOCKS`/`_SEND_RECENT`), the fail-closed hybrid seal (503 `reply_not_sealable`), or the group-send branch. Only the 1:1 Lumina brain-reply tail changes.
- The brain call already runs via `await asyncio.to_thread(brain.reply, ...)` under `_BRAIN_LOCK` — keep that.
- Background task must be crash-safe: catch + log + persist a graceful fallback reply; never leave the turn with no reply and never raise out of the task.
- Line length 99; run tests from `~`.
